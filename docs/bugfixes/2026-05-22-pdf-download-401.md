# Bug Report: Download de PDF retornando 401 Unauthorized

**Data:** 2026-05-22  
**Severidade:** Alta — funcionalidade de exportação inacessível via interface  
**Status:** Corrigido

---

## Descrição

Ao clicar no botão "⬇ PDF" na tela de detalhes de uma teleconsultoria, o endpoint `GET /api/teleconsultorias/{id}/export/pdf` retornava **401 Unauthorized**, mesmo com o usuário autenticado na aplicação.

No Swagger, o mesmo endpoint funcionava corretamente ao preencher manualmente o token Bearer.

---

## Causa Raiz

O template Angular usava um elemento `<a>` com binding de `href`:

```html
<a [href]="pdfUrl()" target="_blank" class="btn btn-secondary btn-sm">⬇ PDF</a>
```

onde `pdfUrl()` retornava apenas a URL string:

```typescript
exportPdfUrl(id: string): string {
  return `${this.base}/${id}/export/pdf`;
}
```

Quando o browser navega por um `<a href>`, ele emite uma requisição HTTP nativa — **fora do controle do Angular**. O `AuthInterceptor` (que injeta o header `Authorization: Bearer <token>`) só intercepta requisições feitas via `HttpClient`. Requisições de navegação direta do browser não passam pelo interceptor, logo o token nunca é enviado.

O backend tem `[Authorize]` no endpoint de exportação, rejeita requisições sem token com 401.

---

## Solução

### Backend
Nenhuma alteração necessária. O endpoint permanece protegido com `[Authorize]` — comportamento correto.

### Frontend — `TeleconsultoriaService`

Substituído `exportPdfUrl()` por `exportPdf()` que usa `HttpClient` com `responseType: 'blob'`:

```typescript
// Antes
exportPdfUrl(id: string): string {
  return `${this.base}/${id}/export/pdf`;
}

// Depois
exportPdf(id: string) {
  return this.http.get(`${this.base}/${id}/export/pdf`, { responseType: 'blob' });
}
```

Com `HttpClient`, a requisição passa pelo `AuthInterceptor` que injeta o token automaticamente.

### Frontend — `DetailComponent`

Substituído `pdfUrl()` por `downloadPdf()` que recebe o Blob e simula download programático:

```typescript
// Antes
pdfUrl(): string {
  return this.tcService.exportPdfUrl(this.id);
}

// Depois
downloadPdf() {
  this.tcService.exportPdf(this.id).subscribe(blob => {
    const url = URL.createObjectURL(blob);
    const a = document.createElement('a');
    a.href = url;
    a.download = `teleconsultoria-${this.id}.pdf`;
    a.click();
    URL.revokeObjectURL(url);
  });
}
```

### Frontend — Template

```html
<!-- Antes -->
<a [href]="pdfUrl()" target="_blank" class="btn btn-secondary btn-sm">⬇ PDF</a>

<!-- Depois -->
<button (click)="downloadPdf()" class="btn btn-secondary btn-sm">⬇ PDF</button>
```

## Teste

Adicionado `teleconsultoria.service.spec.ts` cobrindo o comportamento:

```typescript
it('should request PDF via HttpClient (not direct URL) so auth interceptor adds token', () => {
  const id = 'abc-123';
  const fakeBlob = new Blob(['%PDF'], { type: 'application/pdf' });
  let result: Blob | undefined;

  service.exportPdf(id).subscribe(blob => (result = blob));

  const req = http.expectOne(`${environment.apiUrl}/teleconsultorias/${id}/export/pdf`);
  expect(req.request.method).toBe('GET');
  expect(req.request.responseType).toBe('blob');
  req.flush(fakeBlob);

  expect(result).toBeInstanceOf(Blob);
});
```

O teste verifica que a requisição passa pelo `HttpClient` (e portanto pelo interceptor) e que o `responseType` é `blob`.

---

## Lição

Downloads autenticados **nunca devem usar `<a href>` direto** com URLs de APIs protegidas. Sempre usar `HttpClient` + `URL.createObjectURL()` para que o interceptor de autenticação atue na requisição.

Padrão alternativo válido: passar o token via query string (`?access_token=...`) e validar no backend — já usado no SignalR Hub deste projeto. Porém expõe o token em logs de servidor e histórico do browser; o padrão com `HttpClient` é preferível para endpoints HTTP comuns.
